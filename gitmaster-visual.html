Git Basics
What is Git?
A distributed version control system that tracks changes in source code during software development.
Core Concept

Git is a distributed version control system (DVCS) created by Linus Torvalds in 2005. Unlike centralized systems, every developer has a full copy of the entire repository history on their machine.

Git tracks changes as a series of snapshots — not diffs. Every time you commit, Git takes a picture of what all your files look like at that moment.

Think of Git like a save system in a video game. Every time you save (commit), it records the complete state of your game world. You can always load any previous save — even if you've made hundreds of mistakes since then.

Git Basics
Repository
The container that holds all your project's files, history, and metadata inside a hidden .git folder.
What is a Repository?

A Git repository (repo) is a directory that Git has been initialized in. It contains a hidden .git/ folder where Git stores everything — all your commits, branches, configurations, and object database.

Think of a repository like a book with a very thorough appendix. The book (your code files) is what you read. The appendix (.git folder) contains the complete history, index, and metadata that makes version control possible.
The .git Folder Structure
.git directory
# .git/ structure (created by git init) .git/ ├── HEAD # Points to current branch ├── config # Repository configuration ├── description # Used by GitWeb only ├── hooks/ # Lifecycle hook scripts ├── info/ # Global excludes ├── objects/ # All Git objects (blobs, trees, commits) │ ├── info/ │ └── pack/ └── refs/ # Branch and tag pointers ├── heads/ # Local branch refs ├── remotes/ # Remote tracking refs └── tags/ # Tag refs
Types of Repositories
Local Repository

Lives on your machine. Contains your working files + full history. You can work completely offline.

Remote Repository

Hosted on a server (GitHub, GitLab, Bitbucket). Shared with team. Used for collaboration and backup.

Git Basics
git init
Initialize a new Git repository in the current directory, creating the hidden .git folder.
Command
# Initialize a new repo in current directory git init # Initialize with a specific name (creates new folder) git init my-project # Initialize as a bare repo (for servers) git init --bare repo.git
Running git init is like placing a surveillance camera in your room — from this moment on, Git watches everything. The first time you do this, nothing is recorded yet, but the system is active and ready.

Step-by-Step Internal Working
  • Git creates the .git/ directory in your project folder
  • Creates .git/HEAD pointing to refs/heads/main (default branch)
  • Creates empty .git/objects/ — no commits yet, so the object database is empty
  • Creates .git/refs/heads/ and .git/refs/tags/ — also empty
  • Creates .git/config with default repository settings
  • Your working files are still untracked — Git sees them but hasn't recorded them
Common Mistakes
⚠️ Watch Out
  • Running git init inside another git repo creates a nested repo — usually a mistake
  • Deleting the .git/ folder destroys ALL history permanently
  • Don't commit the .git/ folder itself to any repo

Git Basics
git add
Move changes from the working directory to the staging area (index), preparing them for a commit.
Commands
# Stage a specific file git add index.html # Stage all changes in current directory git add . # Stage all tracked and untracked files everywhere git add -A # Interactively choose which changes to stage (patch mode) git add -p file.js
The staging area is like a shopping cart 🛒. You browse your store (working directory) and put items in your cart (staging). When you checkout (commit), only the items in your cart are purchased — not everything in the store.

What Happens Internally
  • Git reads the file content and computes a SHA-1 hash of it
  • Creates a blob object in .git/objects/ storing the file content
  • Updates the index file (.git/index) to map the filename → blob hash
  • The file is now in "staged" state — ready to be included in the next commit
ℹ️ git add doesn't create a commit — it just prepares the index. You can add and unstage (with git restore --staged) as many times as you want.
Common Mistakes
⚠️ Watch Out
  • git add . stages everything — use .gitignore to exclude node_modules, .env etc.
  • Don't blindly add all files — use git status first to review changes
  • git add -A also stages deletions, while git add . only adds/modifies

Git Basics
git commit
Create a permanent snapshot of staged changes in the repository's history.
Commands
# Standard commit with message git commit -m "Add login feature" # Stage all tracked files and commit in one step git commit -am "Fix bug in auth module" # Amend the most recent commit (change message or add files) git commit --amend -m "Better message" # Empty commit (useful for triggering CI) git commit --allow-empty -m "Trigger CI"
A commit is like taking a photograph 📸 of your project. Every photo has a timestamp, who took it, and what it shows. You can scroll through your photo album (git log) and see exactly how the project looked at any point in time.
Commit Object Structure (Internals)
commit

parent a3f7c2d9... (previous commit)
author John Doe <john@example.com> 1703001600 +0530

Every commit stores: a pointer to a tree (the root directory snapshot), a pointer to the parent commit, author/committer info, and your message. The commit's hash is computed from all of this.

Step-by-Step: What Happens on Commit
  • Git reads the staging area (index) to know what files are staged
  • Creates blob objects for each file's content (if not already stored)
  • Creates tree objects representing the directory structure
  • Creates a commit object pointing to the root tree + parent commit hash
  • Computes SHA-1 hash of the commit object → this is the commit ID
  • Updates the current branch ref (e.g. refs/heads/main) to point to the new commit
  • HEAD still points to the branch, which now points to the new commit
Writing Good Commit Messages
# ✓ Good: subject line + blank + body feat: add OAuth login with Google provider - Implements /auth/google route - Stores tokens in HTTP-only cookies - Adds user session to Redis store Closes #142 # ✗ Bad: vague, no context fix stuff WIP asdfghjkl

Branching
git branch
Branches let you develop features, fix bugs, or experiment in isolated parallel timelines.
Commands
git branch # List local branches git branch -a # List all (local + remote) git branch feature/login # Create a new branch git branch -d feature/login # Delete branch (safe) git branch -D feature/login # Force delete branch git branch -m old-name new-name # Rename branch
A branch is like a parallel universe 🌌 — an exact copy of the main timeline where you can freely experiment. If your experiment succeeds, you merge it back. If not, you just delete the branch — the main timeline is untouched.
What is a Branch Internally?

A branch is just a pointer — a simple file in .git/refs/heads/ containing the hash of the commit it points to. That's it. Creating a branch is almost instant because Git just creates a 41-byte file.

# .git/refs/heads/main contains just: 9d2f3a1b4c5e6d7e8f9a0b1c2d3e4f5a6b7c8d9

Branching
git switch / checkout
Move between branches or create and switch to a new branch in one command.
Commands
# Modern way (git 2.23+) git switch feature/login # Switch to branch git switch -c feature/auth # Create + switch # Classic way (still works) git checkout feature/login git checkout -b feature/auth # Create + switch # Go back to previous branch git switch - # Detach HEAD at a specific commit git checkout a3f7c2d
Switching branches is like switching between parallel universes. When you switch, your working directory files change to match the state of that branch's latest commit. It's instant — Git just updates your files.

Detached HEAD State

When you checkout a specific commit hash (not a branch name), HEAD is "detached" — it points directly to the commit, not through a branch. Any new commits you make here will be lost unless you create a branch!

⚠️ Always create a branch before committing in detached HEAD state: git switch -c my-branch

Branching
git merge
Combine the history of two branches. Git supports fast-forward and three-way merge commits.
Commands
# Merge feature into current branch (main) git merge feature/login # Merge with a commit message always (no fast-forward) git merge --no-ff feature/login # Merge but squash all commits into one git merge --squash feature/login # Abort a merge in progress (during conflicts) git merge --abort
Merging is like writing two stories separately and then combining them into one book. If the stories don't overlap, Git can automatically weave them together. If they conflict (two people edited the same paragraph), you have to decide which version to keep.

Remote Git
git clone
Create a local copy of a remote repository, with all history and branches.
Commands
# Clone from GitHub git clone https://github.com/user/repo.git # Clone into specific folder name git clone https://github.com/user/repo.git my-folder # Clone specific branch only git clone -b develop https://github.com/user/repo.git # Clone with SSH (recommended for push access) git clone git@github.com:user/repo.git # Shallow clone (only last N commits) git clone --depth=1 https://github.com/user/repo.git
git clone is like photocopying an entire book 📚 — you get a complete copy of the remote repository including all chapters (commits), bookmarks (branches), and the index (objects). Your copy is independent: writing in it doesn't change the original.
What Clone Does Internally
  • Creates a new directory with the repo name
  • Initializes a .git/ folder inside it
  • Downloads all objects from the remote (commits, trees, blobs)
  • Configures origin as the remote alias pointing to the URL
  • Creates remote-tracking branches like origin/main
  • Checks out the default branch (usually main) in your working directory

Remote Git
git pull
Download changes from remote AND merge/rebase them into your current branch. Pull = fetch + merge.
Commands

git pull is like receiving an updated newspaper 📰 and reading it immediately. fetch just gets the newspaper delivered, pull also reads it (merges into your branch).
pull = fetch + merge
# These two are equivalent to git pull: git fetch origin git merge origin/main

git fetch

git fetch is like checking your mailbox 📬 — it tells you what's arrived (from remote), but you haven't read the mail yet (haven't merged). It's safe to do at any time; it never changes your local work.

Advanced Git
git rebase
Reapply commits on top of another base tip. Creates a linear history — no merge commits.
Commands

Imagine you're writing a report and discover someone updated the document template while you were writing. Rebase is like copying your changes and re-typing them onto the new template — same content, but now it's based on the latest version.

Advanced Git
git stash
Temporarily save uncommitted changes so you can switch context, then restore them later.
Commands
git stash # Stash tracked changes git stash push -m "WIP: login" # Stash with name git stash -u # Include untracked files git stash list # See all stashes git stash pop # Apply most recent + delete it git stash apply stash@{2} # Apply specific stash (keep it) git stash drop stash@{0} # Delete specific stash git stash clear # Delete ALL stashes git stash branch feature-name # Create branch from stash
Stash is like a desk drawer 🗃️ where you shove all your half-finished work papers when someone walks into your office. When they leave, you pull the papers back out and continue exactly where you left off.
Stash Flow
  • You're working on feature X, half done
  • Urgent bug on main — you need to switch!
  • git stash — saves your WIP, cleans working dir
  • git switch main — now you're on clean main
  • Fix the bug, commit, push
  • git switch feature-x then git stash pop
  • Your WIP is restored exactly as it was

Advanced Git
git reset
Move HEAD (and optionally branch pointer) to a previous commit. Three modes: soft, mixed, hard.
The Three Modes
# --soft: Move HEAD only. Keep staging + working dir. git reset --soft HEAD~1 # Use: "Undo last commit, keep changes staged" # --mixed (default): Move HEAD + unstage. Keep working dir. git reset HEAD~1 # Use: "Undo last commit, keep files but unstage them" # --hard: Move HEAD + unstage + discard working dir changes git reset --hard HEAD~1 # Use: "Completely discard last commit and its changes" # Reset to a specific commit git reset --hard a3f7c2d
git reset --soft is like pressing Ctrl+Z in your editor — the file still shows changes, they're just "undone" in history. --hard is like shredding the paper 🗑️ — the changes are gone.
Reset Modes Comparison
ModeHEAD moves?Staging AreaWorking Dir
--soft✓ YesUnchanged (staged)Unchanged
--mixed✓ YesCleared (unstaged)Unchanged
--hard✓ YesCleared⚠️ Deleted
🚫 Never use --hard on shared branches — it rewrites public history. Use git revert instead to safely undo commits on shared branches.

Advanced Git
git reflog
The safety net. Reflog records every movement of HEAD — your last resort for recovering lost commits.
Commands
git reflog # Show HEAD's history git reflog show main # Show main branch's history # Recover a "lost" commit after bad reset: git reflog # Find the commit hash git reset --hard HEAD@{3} # Go back to it # Or create a branch from the lost commit git checkout -b recovery HEAD@{3}
Reflog is your time machine's flight recorder 🛩️. Even if you git reset --hard and seem to have lost everything, reflog remembers every position HEAD has ever been in, for up to 90 days.
Sample Reflog Output
9d2f3a1 HEAD@{0}: commit: Add login feature f3bc1d2 HEAD@{1}: reset: moving to HEAD~1 a3f7c2d HEAD@{2}: commit: Add auth module b7e9f4c HEAD@{3}: checkout: moving from feature to main c1a2b3d HEAD@{4}: commit: Initial commit

Advanced Git
HEAD
HEAD is a pointer to the currently checked-out commit. It's where you are in the repository right now.
What is HEAD?

HEAD is a special file at .git/HEAD. It usually contains a reference to a branch (attached HEAD), or a direct commit hash (detached HEAD).

# Attached HEAD (normal state) cat .git/HEAD ref: refs/heads/main # Detached HEAD (after git checkout <commit-hash>) cat .git/HEAD a3f7c2d9e1b4f5c6d7e8f9a0b1c2d3e4f5a6b7c8
HEAD Notation
HEAD # Current commit HEAD~1 # Parent of current commit HEAD~2 # Grandparent HEAD^ # Same as HEAD~1 HEAD^^ # Same as HEAD~2 HEAD^2 # Second parent (merge commits only) HEAD@{1} # Previous value from reflog
HEAD is like a "You Are Here" marker 📍 on a map. As you move around (checkout different commits/branches), the marker follows you. When you commit, the marker and the branch pointer both advance together.

Git Internals
Git Object Model & DAG
How Git actually stores data: blobs, trees, commits, and the Directed Acyclic Graph (DAG).
The 4 Object Types
📄 Blob

Stores raw file content (no filename). Identified by SHA-1 hash of content.

blob
console.log("hello");
🌲 Tree

A directory. Contains pointers to blobs (files) and other trees (subdirectories).

tree
100644 blob a3f7c2d index.html
100644 blob b9f2e1c style.css
040000 tree c1d2e3f src/
💾 Commit

Points to a root tree + parent commit. Contains metadata (author, message).

commit
tree 4b825dc
parent a3f7c2d
author Jane <j@ex.com>

Add login feature
🏷️ Tag

A named pointer to a specific commit. Annotated tags store extra metadata.

tag
object 9d2f3a1
type commit
tag v1.0.0
The DAG — Directed Acyclic Graph

Git's history is a DAG — commits point backward to their parents, never in circles. This structure is what makes Git so powerful: branching is free, merging is safe, and history is immutable.

git-dag.viz
Content-Addressed Storage

Git identifies every object by its SHA-1 hash. The hash is computed from the content itself — so identical content always produces the same hash, and Git stores it only once (deduplication is automatic).

# Inspect any object git cat-file -t a3f7c2d # Type: blob/tree/commit/tag git cat-file -p a3f7c2d # Pretty-print content # Manually hash content (see what hash a file would get) git hash-object file.txt # See the tree of a commit git ls-tree HEAD
